2. Case Study (Mô tả bài toán nghiệp vụ)

Lĩnh vực: Bán hàng trực tuyến cho cửa hàng đồ uống (Coffee & Tea Shop).

Bối cảnh: Một chuỗi cửa hàng cà phê – trà sữa muốn có một ứng dụng di động giúp khách hàng tự đặt đồ uống mà không cần xếp hàng. Khách có thể:

  • Tạo tài khoản, đăng nhập để quản lý thông tin cá nhân.
  • Xem menu đồ uống (17 sản phẩm), tìm kiếm và lọc theo loại (Cà phê, Trà, Trà sữa, Đá xay, Sinh tố, Nước ép).
  • Xem chi tiết, chọn size (S/M/L) và số lượng, thêm vào giỏ hàng.
  • Đặt hàng kèm địa chỉ giao và phương thức thanh toán.
  • Xem lại lịch sử các đơn đã đặt.
  • Xem vị trí các cửa hàng trên bản đồ.

Lợi ích: Giảm thời gian chờ, tăng trải nghiệm khách hàng, lưu lại lịch sử mua hàng phục vụ chăm sóc khách hàng.

3. Phân tích nghiệp vụ & Thiết kế hệ thống

3.1. Yêu cầu chức năng (Functional Requirements)

  1. FR1 – Đăng ký tài khoản mới (lưu vào CSDL).
  2. FR2 – Đăng nhập / Đăng xuất.
  3. FR3 – Xem và cập nhật hồ sơ cá nhân, đổi ảnh đại diện bằng ảnh thật (chụp từ máy ảnh hoặc chọn từ thư viện).
  4. FR4 – Hiển thị danh sách sản phẩm từ CSDL.
  5. FR5 – Tìm kiếm và lọc sản phẩm theo loại.
  6. FR6 – Xem chi tiết sản phẩm, chọn size & số lượng.
  7. FR7 – Thêm/sửa/xóa sản phẩm trong giỏ hàng, tự tính tổng tiền.
  8. FR8 – Thanh toán: nhập địa chỉ, chọn phương thức, lưu đơn hàng vào CSDL.
  9. FR9 – Xem lịch sử đơn hàng và cập nhật trạng thái từng đơn (Chờ thanh toán → Đã thanh toán, hoặc Hủy đơn).
  10. FR10 – Xem vị trí cửa hàng trên bản đồ.

3.2. Yêu cầu phi chức năng (Non-functional Requirements)

  • NFR1 – Giao diện thân thiện, đồng bộ (Material Design 3).
  • NFR2 – Dữ liệu lưu tập trung trên SQL Server, truy cập qua REST API (kiến trúc client–server).
  • NFR3 – Phản hồi nhanh, có hiệu ứng loading khi xử lý.
  • NFR4 – Mã nguồn rõ ràng, có comment, dễ bảo trì.
  • NFR5 – Kiểm thử tự động bằng Unit Test & Widget Test.

3.3. Kiến trúc ứng dụng (Architecture)

Ứng dụng theo mô hình client–server 3 lớp: Frontend (Flutter) – Backend (.NET Web API) – Database (SQL Server). Frontend quản lý trạng thái bằng Provider.

+----------------------------+        +---------------------------+        +----------------+
|  FRONTEND (Flutter)        |  HTTP  |  BACKEND (ASP.NET Core)   |  EF    |  DATABASE      |
|  screens / providers /     | -----> |  Controllers + EF Core    | -----> |  SQL Server    |
|  ApiService (http)         |  JSON  |  (Auth/Products/Orders)   |  Core  |  CoffeeShopDb  |
+----------------------------+        +---------------------------+        +----------------+
  • Frontend (Flutter): các màn hình (View) + Provider quản lý trạng thái; lớp ApiService gọi REST API bằng gói http.
  • Provider (State Management): AuthProvider, ProductProvider, CartProvider kế thừa ChangeNotifier, gọi notifyListeners() để UI build lại.
  • Backend (ASP.NET Core Web API): các Controller xử lý nghiệp vụ, dùng Entity Framework Core thao tác dữ liệu.
  • Database (SQL Server): lưu toàn bộ dữ liệu thật (users, products, orders, order_items).

3.5. Công nghệ mới tự tìm hiểu (ngoài bài học)

  • provider – quản lý trạng thái theo mẫu ChangeNotifier.
  • flutter_map + OpenStreetMap – hiển thị bản đồ miễn phí, không cần API key (thay cho Google Maps phải tính phí).
  • intl – định dạng tiền tệ VND và ngày giờ.
  • image_picker – chọn ảnh từ thư viện hoặc chụp ảnh bằng máy ảnh để làm ảnh đại diện.
  • path_provider – lấy thư mục riêng của app để lưu ảnh đại diện lâu dài, không bị xóa.
  • http – gọi REST API (HTTP/JSON) từ Flutter tới backend.
  • url_launcher – mở Google Maps để chỉ đường tới cửa hàng.
  • ASP.NET Core Web API (.NET) – xây dựng backend cung cấp REST API.
  • Entity Framework Core – ORM ánh xạ đối tượng C# sang bảng SQL Server, thao tác dữ liệu không cần viết SQL thủ công.

3.7. Thiết kế Backend API (.NET)

Backend là một dự án ASP.NET Core Web API kết nối SQL Server qua Entity Framework Core, cung cấp các điểm cuối (endpoint):

Phương thứcĐường dẫnChức năng
POST/api/auth/registerĐăng ký tài khoản
POST/api/auth/loginĐăng nhập
PUT/api/auth/{id}Cập nhật hồ sơ + avatar
GET/api/productsLấy danh sách sản phẩm
GET/api/orders?userId=Lịch sử đơn của người dùng
POST/api/ordersTạo đơn hàng mới
PUT/api/orders/{id}/statusCập nhật trạng thái đơn

App Flutter gọi các endpoint này qua lớp ApiService; dữ liệu trả về dạng JSON và được lưu thật trong SQL Server.

4. Yêu cầu phát triển (Development)

  • UI Implementation: 9 màn hình theo Material Design 3, theme tông cà phê đồng bộ, sử dụng ảnh thật của đồ uống (đóng gói trong assets để chạy offline) và ảnh đại diện do người dùng tự chọn.

6. Kết luận & Thảo luận

Ưu điểm:

  • Đầy đủ luồng mua hàng end-to-end theo kiến trúc client–server (Flutter + .NET + SQL Server).
  • Áp dụng đúng Provider cho frontend và Entity Framework Core cho backend; mã nguồn rõ ràng có comment.
  • Giao diện đẹp, đồng bộ; dữ liệu lưu tập trung trên SQL Server; có kiểm thử tự động.
  • Dùng ảnh thật cho sản phẩm và avatar; bản đồ có chỉ đường tới cửa hàng.
  • Menu phong phú (17 sản phẩm, 6 loại); quản lý vòng đời đơn hàng theo trạng thái (Chờ thanh toán / Đã thanh toán / Đã hủy).

Nhược điểm:

  • Mật khẩu lưu dạng plain text (chưa mã hóa/băm).
  • Chưa có xác thực bằng token (JWT); API chưa phân quyền.
  • Chưa có vai trò quản trị (admin) để thêm/sửa sản phẩm.

Bài học rút ra:

  • Hiểu cách quản lý trạng thái bằng Provider và vòng đời widget.
  • Biết xây dựng REST API bằng ASP.NET Core + Entity Framework Core và kết nối SQL Server.
  • Hiểu cách frontend (Flutter) giao tiếp với backend qua HTTP/JSON.
  • Biết viết Unit Test / Widget Test và build APK Release.

Hướng phát triển nếu có thêm thời gian:

  • Băm mật khẩu (hash) và thêm xác thực JWT cho API.
  • Triển khai backend lên server/đám mây (Azure) để dùng từ xa.
  • Thêm trang quản trị, thanh toán online thật, thông báo đẩy.
