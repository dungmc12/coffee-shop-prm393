2. Case Study

Domain: Online sales for a beverage shop (Coffee & Tea Shop).

Context: A chain of coffee and milk-tea shops wants a mobile application that lets customers order drinks by themselves without queuing. Customers can:

  • Create an account and log in to manage personal information.
  • Browse the drink menu (17 products), search and filter by category (Coffee, Tea, Milk Tea, Blended, Smoothie, Juice).
  • View product details, choose size (S/M/L) and quantity, and add to the cart.
  • Place an order with a delivery address and payment method.
  • Review the history of placed orders.
  • View the locations of the shops on a map.

Benefits: Reduced waiting time, improved customer experience, and stored purchase history for customer care.

3.3. Application Architecture

The application follows a 3-tier client–server model: Frontend (Flutter) – Backend (.NET Web API) – Database (SQL Server). The frontend manages state with Provider.

+----------------------------+        +---------------------------+        +----------------+
|  FRONTEND (Flutter)        |  HTTP  |  BACKEND (ASP.NET Core)   |  EF    |  DATABASE      |
|  screens / providers /     | -----> |  Controllers + EF Core    | -----> |  SQL Server    |
|  ApiService (http)         |  JSON  |  (Auth/Products/Orders)   |  Core  |  CoffeeShopDb  |
+----------------------------+        +---------------------------+        +----------------+
  • Frontend (Flutter): UI screens + Provider for state; an ApiService layer calls the REST API using the http package.
  • Provider (State Management): AuthProvider, ProductProvider, CartProvider extend ChangeNotifier and call notifyListeners() to trigger UI rebuilds.
  • Backend (ASP.NET Core Web API): Controllers handle business logic and use Entity Framework Core for data access.
  • Database (SQL Server): stores all real data (users, products, orders, order_items).

3.5. New Technologies Learned (outside class)

  • provider – state management using the ChangeNotifier pattern.
  • flutter_map + OpenStreetMap – displays a free map with no API key required (instead of paid Google Maps).
  • intl – formats currency (VND) and date/time.
  • image_picker – picks an image from the gallery or takes a photo with the camera to use as an avatar.
  • path_provider – locates the app's private directory to persist the avatar image so it is not deleted.
  • http – calls the REST API (HTTP/JSON) from Flutter to the backend.
  • url_launcher – opens Google Maps to get directions to a shop.
  • ASP.NET Core Web API (.NET) – builds the backend that exposes the REST API.
  • Entity Framework Core – an ORM that maps C# objects to SQL Server tables, handling data without writing manual SQL.

3.7. Backend API Design (.NET)

The backend is an ASP.NET Core Web API project connected to SQL Server through Entity Framework Core, exposing the following endpoints:

MethodRouteFunction
POST/api/auth/registerRegister an account
POST/api/auth/loginLog in
PUT/api/auth/{id}Update profile + avatar
GET/api/productsGet the product list
GET/api/orders?userId=A user's order history
POST/api/ordersCreate a new order
PUT/api/orders/{id}/statusUpdate an order's status

The Flutter app calls these endpoints through the ApiService class; responses are JSON and the data is stored for real in SQL Server.

6. Conclusion & Discussion

Strengths:

  • Complete end-to-end purchase flow with a client–server architecture (Flutter + .NET + SQL Server).
  • Correctly applies Provider on the frontend and Entity Framework Core on the backend; clean, commented code.
  • Beautiful, consistent UI; data stored centrally on SQL Server; includes automated testing.
  • Uses real photos for products and avatars; the map provides directions to shops.
  • Rich menu (17 products, 6 categories); manages the order lifecycle by status (Pending payment / Paid / Cancelled).

Weaknesses:

  • Passwords are stored as plain text (not hashed).
  • No token-based authentication (JWT); the API has no authorization yet.
  • No admin role to add/edit products.

Lessons learned:

  • Understanding state management with Provider and the widget lifecycle.
  • Building a REST API with ASP.NET Core + Entity Framework Core connected to SQL Server.
  • How a Flutter frontend communicates with a backend over HTTP/JSON.
  • Writing Unit Tests / Widget Tests and building a Release APK.

Future improvements (with more time):

  • Hash passwords and add JWT authentication for the API.
  • Deploy the backend to a server/cloud (Azure) for remote access.
  • Add an admin page, real online payment, and push notifications.
